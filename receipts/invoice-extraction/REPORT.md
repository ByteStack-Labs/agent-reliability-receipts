## Summary

The extractor is **confidently, silently wrong** — not uncertain. It scores
**100% exact match on evaluation and 86.25% in production**, a 13.75-point drop.
The whole drop comes from one cause: the extractor assigns each value by its
**line position**, not by the label beside it. On records whose field order or
line structure shifts, it transposes values into the wrong fields and still
returns clean, fully-populated output. **49 of 400 production records (12.25%)
are wrong while structurally indistinguishable from a correct record** — no error
raised, no empty field, no signal anything failed. The headline is not the 14
points lost; it is that most of what is lost is lost silently.

## The gap

**What evaluation measured.** 400 invoices in a single canonical format: eleven
fields, one per line, fixed order, `Label: value`. Metric: exact match (all 11
fields correct) and mean field accuracy. Result: 1.0 / 1.0.

**What deployment requires.** The same eleven fields arriving in real-world
formatting variety — different delimiters (`|`, `=>`, tab, comma), extra
whitespace, reordered fields, single-line records, decorative header lines. The
metric that would have caught it is **exact match sliced by format**, not the
aggregate.

The eval set exercises exactly one of the six formats production produces. The
investigation is that one-format-vs-six gap.

## Reproduction

Minimal reproducing set — same content, formatting only differs:

- **Eval-style (passes):** `"Invoice: INV-10000\nVendor: ...\nClient: ..."`
  → exact_match = 1.
- **Production-style, one adjacent swap (fails):** the same record with the
  `Vendor` and `Client` lines exchanged → exact_match = 0, and **exactly those
  two fields are wrong, transposed into each other.** Nothing else changes.

`verify.py` section 4 constructs both from a single ground-truth record and
confirms the control passes and the swap fails. This is the smallest input that
elicits the failure: one swapped adjacent pair.

is entirely the field-moving slices, each at 0.000, masked by the order-

**Confidence / silent-failure finding.** The extractor exposes **no confidence
signal** — `mean_field_accuracy` is a measured score, not something the system
emits about itself, and the report does not relabel it as confidence. So the
calibration question becomes: when wrong, does anything about the output betray
it? Of the 55 wrong records:

- **49 are silent** (well-formed, all 11 fields populated): `reorder` 36,
  `mixed` 5, `verbose` 8. **12.25% of all traffic is wrong with zero signal.**
- **6 are detectable** (an empty field a null-check would flag): all `compact`,
  where the single-line collapse leaves 10 fields empty.

The detectable 6 are caught by accident, not design, and they mask the problem:
a downstream null-check would pass on every one of the 49 that matter. This is
the confidently-wrong signature — correctness collapses while the output's self-
evidence of correctness does not move.

## Root cause

**Mechanism: positional field assignment.** `extract` zips the field list against
the input lines by index (`out[field] = value_after_label(lines[i])`), so it
reads *where* a value sits, never *what* it is labeled. Value parsing
(`_value_after_label`) is robust — it handles every delimiter — which is why
order-preserving shifts survive. The fragility is entirely the index binding.

**Ablations (each isolates one variable):**

- **Ablation A — single adjacent swap.** Hold content, delimiter, and line count
  constant; exchange only the `Vendor` and `Client` lines. Result: exact_match
  drops 1 → 0, and **exactly those two fields are wrong, with their values
  transposed** (`vendor` gets the client's value and vice-versa). A label-reading
  extractor would be unaffected; a position-reading one fails precisely here.
  Evidence strength: **decisive** — one variable changed, predicted effect
  observed exactly.
- **Ablation B — one decorative leading line.** Prepend `=== INVOICE ===`;
  labels and order untouched. Result: **all 11 fields wrong**; `invoice_id` reads
  the banner. One extra line shifts every position by one — an off-by-one cascade
  that only a positional reader can produce.

Measured, not inferred: the gap, the slices, the silent/detectable split, and
both ablations are all computed by `verify.py`. Inferred (low risk): that the
live production format mix matches the synthetic weighting here — this is a
labeled teaching fixture, so the mix is stipulated, not sampled from real
traffic.

## Mechanistic corroboration

The behavioral diagnosis is already decisive, and corroboration agrees: per-field
error attribution under Ablation A shows error localized to exactly the moved
indices (field accuracy 0.8182 = 9/11 correct, the 2 swapped wrong), and under
Ablation B error is total (0.0000) — the two signatures of index-based reading
(local swap vs. global shift). No model internals exist to inspect; none are
needed.

## Open questions

- **Live format distribution.** The 187/158/36/8/6/5 shift mix is stipulated by
  the fixture, not measured from production traffic. On a real system the next
  step is to sample actual inputs and recompute the per-slice rates; the silent
  error rate scales directly with the field-moving share.
- **Partial-credit fields.** This autopsy scores exact match and field accuracy.
  If some fields matter more than others (e.g. `total` vs `item`), a value-weighted
  metric would re-rank severity; not measured here.
- **Fix validation.** The remediation (bind values to labels) is named but, per
  methodology, not implemented or measured in this diagnostic. Validating it would
  mean re-running all six slices and confirming field-moving rates rise to 1.000.
