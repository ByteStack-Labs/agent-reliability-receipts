# invoice-extraction Production Autopsy

> Generated by running `/agent-reliability:production-autopsy` against
> `fixtures/invoice-extraction`. Every number below is reproduced from the
> committed fixture artifacts by `verify.py` (standard library, seeded, no model,
> no GPU). From this directory, run `python3 verify.py` to regenerate them; the
> script exits non-zero if any figure fails to reproduce. The committed `data/`
> and `results/` are treated as immutable ground truth and were measured as-is —
> nothing was regenerated.

## Summary

The extractor is **confidently, silently wrong** — not uncertain. It scores
**100% exact match on evaluation and 86.25% in production**, a **13.75-point
drop**. The entire drop traces to one cause: the extractor assigns each value by
its **line position**, not by the label beside it. On records whose field order
or line structure shifts, it transposes values into the wrong fields and still
returns clean, fully-populated output. **49 of 400 production records (12.25%)
are wrong while structurally indistinguishable from a correct record** — no error
raised, no empty field, no signal anything failed. The headline is not the ~14
points lost; it is that most of what is lost is lost *silently*.

## The gap

**What evaluation measured.** 400 invoices in a single canonical format: eleven
fields, one per line, fixed order, `Label: value`. Metric: exact match (all 11
fields correct) and mean field accuracy. Result: **1.0 / 1.0**.

**What deployment requires.** The same eleven fields arriving in real-world
formatting variety — different delimiters (`|`, `=>`, tab, comma), extra
whitespace, reordered fields, single-line records, decorative header lines. The
metric that would have caught the failure is **exact match sliced by format**,
not the aggregate.

The eval set exercises exactly **one** of the six formats production produces.
That one-format-vs-six divergence is the entire investigation.

## Reproduction

Minimal reproducing set — same content, only formatting differs:

- **Eval-style (passes):** `Invoice: INV-10000` / `Vendor: …` / `Client: …` (one
  field per line, fixed order) → `exact_match = 1`.
- **Production-style, one adjacent swap (fails):** the same record with the
  `Vendor` and `Client` lines exchanged → `exact_match = 0`, and **exactly those
  two fields are wrong, transposed into each other** (`vendor` receives the
  client's value and vice-versa). Nothing else changes.

`verify.py` section 4 builds both from a single committed ground-truth record and
confirms the control passes and the swap fails. One swapped adjacent pair is the
smallest input that elicits the failure.

## Measurements

Re-scored from raw text in `fixtures/invoice-extraction/data/*.jsonl` (not read
back from `results/`):

| Split | n | Exact match | Mean field accuracy |
|-------|---|-------------|---------------------|
| eval | 400 | **1.0000** | 1.0000 |
| production | 400 | **0.8625** | 0.9464 |

Aggregate exact-match drop: **13.75 points**.

**Per-slice (the slice is the whole story).** The aggregate hides a clean binary:

| shift_type | n | exact match | class |
|------------|---|-------------|-------|
| delimiter | 187 | **1.000** | order-preserving |
| whitespace | 158 | **1.000** | order-preserving |
| reorder | 36 | **0.000** | field-moving |
| verbose | 8 | **0.000** | field-moving |
| compact | 6 | **0.000** | field-moving |
| mixed | 5 | **0.000** | field-moving |

Field-moving records = 36 + 8 + 6 + 5 = **55/400 = 13.75%**, which equals the
production error rate **exactly**. The drop is not spread across the
distribution; it is entirely the field-moving slices, each at 0.000, masked by
the order-preserving slices at 1.000.

**Confidence / silent-failure finding.** The extractor exposes **no confidence
signal**. `mean_field_accuracy` is a measured score against ground truth, not
something the system emits about itself, so it is never relabeled as confidence.
The calibration question therefore becomes: when wrong, does anything about the
output betray it? Of the 55 wrong records:

- **49 are silent** — well-formed, all 11 fields populated: `reorder` 36,
  `mixed` 5, `verbose` 8. **12.25% of all traffic is wrong with zero signal.**
- **6 are detectable** — an empty field a null-check would flag: all `compact`,
  where the single-line collapse leaves fields empty.

The detectable 6 are caught by accident, not design, and they *mask* the problem:
a downstream null-check passes on every one of the 49 that matter. This is the
confidently-wrong signature — correctness collapses while the output's
self-evidence of correctness does not move.

## Root cause

**Mechanism: positional field assignment.** `extract` zips the field list against
the input lines by index (`out[field] = _value_after_label(lines[i])`), so it
reads *where* a value sits, never *what* it is labeled. Value parsing
(`_value_after_label`) is robust — it handles every delimiter — which is exactly
why order-preserving shifts (delimiter, whitespace) survive at 1.000. The
fragility is entirely the index binding.

**Ablations (each isolates one variable):**

- **Ablation A — single adjacent swap.** Hold content, delimiter, and line count
  constant; exchange only the `Vendor` and `Client` lines. Result: exact_match
  `1 → 0`, field accuracy `1.0 → 0.8182` (9/11), and **exactly those two fields
  are wrong, with their values transposed**. A label-reading extractor would be
  unaffected; a position-reading one fails precisely here. Evidence strength:
  **decisive** — one variable changed, predicted effect observed exactly.
- **Ablation B — one decorative leading line.** Prepend `=== INVOICE ===`;
  labels and order untouched. Result: **all 11/11 fields wrong** — a single extra
  line shifts every position by one, and `invoice_id` absorbs the banner text.
  Confirms the binding is to absolute line index, not content.

**Measured vs. inferred.** Measured: the per-slice 0.000/1.000 split, the
55/400 field-moving fraction equalling the error rate, both ablation outcomes,
and the 49/6 silent/detectable split. Inferred (but strongly supported): that
live production traffic will exhibit the same failure wherever its format mix
includes field-moving variants — the rate depends on that mix, which this fixture
sets by a fixed weighting rather than measuring from real logs.

## Mechanistic corroboration

Not applicable in the model sense — there is no model, no attention or activation
to inspect. The mechanism is corroborated directly at the code level: the single
line `out[field] = _value_after_label(lines[i])` in `extractor.py` is the index
binding, and Ablation B's off-by-one cascade (one extra line → every field wrong)
is the behavioral fingerprint of absolute-position indexing.

## Open questions

- **Live format mix.** The 13.75% production error rate is exact *for this
  fixture's* shift weighting. The real number depends on how often field-moving
  formats occur in production traffic, which requires sampling live inputs. What
  would resolve it: a labelled sample of real invoice formats, sliced the same
  way.
- **Are there field-moving formats beyond these four?** The fixture enumerates
  reorder/mixed/compact/verbose. Production may contain shapes (multi-line
  values, missing fields, merged lines) not represented here. A coverage audit of
  live-vs-eval formats would close this.
- **Detection design.** A per-field provenance signal (which line/label each
  value came from) would convert the 49 silent failures into flaggable ones. Its
  precision/recall against ground truth is unmeasured until such a signal exists.
